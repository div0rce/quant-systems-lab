Setup: one publisher bursts every market-data datagram back-to-back (no pacing); one
subscriber drains them and reports how many it received. The only variable is the
subscriber's requested SO_RCVBUF. Requested 0 = OS default; the kernel may round up or
clamp the request, so the effective (granted) size is read back via getsockopt.

Reading the result: 'lost/trial' is the honest loss metric -- published minus received
-- so it counts TAIL drops too. 'seq-gaps/trial' is the SequenceTracker count, which
sees only INTERIOR gaps: a datagram dropped at the very end of the burst has no later
datagram to reveal a missing sequence number, so a trial can show real loss with zero
sequence gaps. A too-small receive buffer overflows under the burst and the kernel drops
datagrams; the OS default and larger buffers absorb the same burst with little or no
loss. Loss is timing/OS/load dependent, so per-trial counts vary run-to-run -- the
mechanism (SO_RCVBUF bounds in-kernel queueing) is the point, not a fixed number.

Caveats:
- Loopback only: no NIC, driver, routing, or real-network loss is exercised.
- This measures the receive socket-buffer / kernel-drop mechanism, not throughput,
  not latency, and not a production capacity claim.
- No retransmit channel exists; loss is detected (gap count), never recovered.
