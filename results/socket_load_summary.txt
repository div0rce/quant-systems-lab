Reading the result: compare how the best wall time grows with the client count within each
mode. In principle the blocking server (one connection at a time) should scale worse than
epoll (which multiplexes), but at these small loopback counts connection setup dominates and
the two modes stay close (same order of magnitude); which is marginally faster varies run to
run. A clear epoll advantage would require higher concurrency or heavier per-connection work.
Absolute conns/s figures are loopback, single-machine, and load dependent -- not a
production-capacity or throughput claim. Every 'completed' cell is N/N: a load run that
cannot reach full completion writes no artifact and fails, so this table is never partial
unless QSL_LOAD_ALLOW_PARTIAL=1 was set.

Caveats:
- Loopback only: no NIC, driver, routing, or real-network behaviour is exercised.
- Each client is a brand-new short-lived connection (connection-setup heavy), not a
  long-lived high-throughput session; this measures connection scaling, not steady-state
  message throughput.
- Spawning client processes adds fork/exec cost on the driver side; both modes pay it
  equally, so the blocking-vs-epoll comparison is still meaningful, but absolute conns/s
  is bounded by client spawn cost, not just the server.
- Constrained-environment evidence for investigation; not a production-capacity claim.
