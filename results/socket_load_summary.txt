Reading the result: compare how the best wall time grows with the client count within
each mode. The blocking server serializes concurrent connections while epoll multiplexes
them, so epoll should grow more slowly as N rises -- but at these small loopback counts,
where per-connection serve work is tiny and connection setup dominates, the two modes can
look similar (same order of magnitude); the epoll advantage only emerges once
serialization dominates. Absolute conns/s figures are loopback, single-machine, and load
dependent -- not a production-capacity or throughput claim. The 'completed' column shows
the worst per-trial completion; anything below N/N means a trial dropped connections.

Caveats:
- Loopback only: no NIC, driver, routing, or real-network behaviour is exercised.
- Each client is a brand-new short-lived connection (connection-setup heavy), not a
  long-lived high-throughput session; this measures connection scaling, not steady-state
  message throughput.
- Spawning client processes adds fork/exec cost on the driver side; both modes pay it
  equally, so the blocking-vs-epoll comparison is still meaningful, but absolute conns/s
  is bounded by client spawn cost, not just the server.
- Transient: under rapid gateway start/stop cycling a trial can occasionally fail to bind
  or serve on a constrained host, shown as a '< N/N' completed cell; the reported best
  wall time is taken only from trials whose gateway actually served (failed trials do not
  poison the minimum).
- Constrained-environment evidence for investigation; not a production-capacity claim.
